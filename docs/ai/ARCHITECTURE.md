# Architecture - Will Tech OrbitFlow

> Backend-first SaaS foundation. Keep the core clean, secure, and adaptable.

## High-Level Overview

```text
Clients
(web app, mobile app, admin app, AI-generated frontend)
    |
    v
NGINX API Gateway
(rate limiting, headers, proxy)
    |
    v
apps/api
NestJS REST API
    |
    |-- PostgreSQL  # primary relational data
    |-- Redis       # refresh tokens, cache, future queues
    `-- Events      # domain hooks for async workflows
```

OrbitFlow does not ship with a required frontend. A frontend can be added later in `apps/web`, deployed separately, or generated by an AI assistant using the guides in `docs/FRONTEND_GUIDE.md` and `docs/ai/FRONTEND_APP_PROMPT.md`.

## API Module Architecture

Feature modules should follow this shape when useful:

```text
src/modules/<feature>/
|-- <feature>.module.ts
|-- <feature>.controller.ts
|-- <feature>.service.ts
|-- <feature>.repository.ts
|-- dto/
|   |-- create-<feature>.dto.ts
|   `-- update-<feature>.dto.ts
`-- <feature>.service.spec.ts
```

Current implemented modules:

- `auth`
- `users`
- `workspaces`
- `boards`
- `posts`
- `votes`
- `comments`
- `health`

## Request Flow

```text
Request
  -> Global guards
  -> DTO validation
  -> Controller
  -> Service
  -> Repository
  -> Prisma
  -> PostgreSQL
```

Side effects should be emitted through events first. Workers, queues, and notifications can be added later without forcing the first version into microservices.

## Security Flow

```text
Request
  -> ThrottlerGuard
  -> JwtAuthGuard
  -> WorkspaceAccessService role checks
  -> Feature service
```

Rules:

- Routes are private by default.
- Use `@Public()` only for routes that truly need no JWT.
- Workspace-owned operations must validate tenant access.
- Shared mutation endpoints should require role checks.

## Multi-Tenancy Model

Core tenant model:

- `Workspace`
- `WorkspaceMember`
- workspace-scoped roles: `OWNER`, `ADMIN`, `MEMBER`, `VIEWER`
- workspace-owned resources such as boards and posts

Tenant access is centralized in:

```text
apps/api/src/common/authorization/workspace-access.service.ts
```

Do not duplicate tenant authorization logic inside frontend code. The frontend may hide buttons based on role, but the backend must enforce access.

## Frontend Strategy

Frontend is intentionally optional.

Recommended placement when needed:

```text
apps/web
```

The frontend must respect:

- API base path: `/api/v1`
- success envelope: `{ data, meta }`
- error envelope: `{ error }`
- bearer token authentication
- workspace context
- backend-owned authorization

See:

- `docs/FRONTEND_GUIDE.md`
- `docs/frontend/ADD_FRONTEND.md`
- `docs/ai/FRONTEND_APP_PROMPT.md`

## Environment Strategy

| Environment   | Purpose               | Database         |
| ------------- | --------------------- | ---------------- |
| `development` | Local dev with Docker | Local PostgreSQL |
| `staging`     | Pre-production tests  | Staging DB       |
| `production`  | Live deployment       | Production DB    |

## Scaling Direction

Start as a modular monolith.

Extract only when pressure is real:

- background workers;
- notification service;
- websocket service;
- billing integration;
- analytics pipeline.

The goal is not to over-engineer early. The goal is to keep boundaries clear enough that scaling remains possible.
